# Phase 4 P0 Implementation Readiness Gate — R1
**🟡 Gemini (系统环境/LLM 风险/长期可维护性)**

### 1. Go/No-Go
**Go.** 核心结构依赖已在 Migration 0011 中补齐，特别是 P0-0 (fact_links) 的提升解决了 P0-3 (graph_health) 的逻辑悬空问题。P0-7 (backup) 的加入为本阶段的高频数据变动提供了必要的安全边界。

### 2. Top 3 风险 (系统/环境/长期视角)
- **风险 1：系统自消耗循环 (Self-Consumption Loop)**
    - **触发**：LLM 生成的 Wiki 页面被 ingest 逻辑错误识别为原始事实（source chunks）并重新存入 ledger。
    - **损失**：信息在系统内发生正反馈循环，导致原始数据纯度受损，最终造成知识库不可用。
    - **预防**：执行 P0-6 中的来源黑名单机制，确保 extractor 拒绝处理系统自身生成的路径。
- **风险 2：递归查询性能爆炸导致 Daemon 挂起**
    - **触发**：事实图 (fact_links) 密度过高，且 `triage` 或 `reflect` 运行深度优先搜索时。
    - **损失**：终端查询延迟激增，Daemon 线程被 SQL 锁死，导致用户输入丢失或 CLI 卡顿。
    - **预防**：在所有递归查询中硬编码深度限制，并为 `triage` 扫描增加 `LIMIT`。
- **风险 3：修正检测 (Correction) 的启发式偏见**
    - **触发**：Regex 误判用户在对话中的转折词，将其误认为对存量事实的否定。
    - **损失**：高置信度事实被错误标记为错误，系统稳定性受损。
    - **预防**：坚持 P0-5 的“信号化”原则，所有检测仅作为 triage 信号，严禁自动降低 facts 置信度。

### 3. P0 顺序裁决
认可 synthesis 提议，建议将 **P0-6 (LLM 安全)** 保持在 Week 2，以防逻辑触发 Wiki 重建导致数据污染：
- **Week 1**: **P0-0** (fact_links 基础设施) + **P0-7** (数据安全备份)
- **Week 2**: **P0-6** (LLM 降级与自消耗防护) + **P0-4** (archive_reason 写入)
- **Week 3**: **P0-3** (graph_health 实现) + **P0-5** (纠错信号捕获)
- **Week 4**: **P0-2** (决策审计) + **P0-1** (Triage 综合看板)

### 4. 测试盲点 (≥ 2)
1. **LLM 故障注入测试 (Fault Injection)**: 模拟 LLM 离线或返回畸形数据时提取器的行为，验证 P0-6 的 Circuit Breaker。
2. **Schema 损坏恢复测试**: 验证在 Ledger 损坏的情况下能否通过 P0-7 的备份机制完全恢复。

### 5. 撤退条件 (≥ 2)
1. **备份还原一致性校验失败**: 如果 P0-7 无法在模拟故障后恢复一致状态，应立即停止所有涉及数据写回的任务。
2. **Wiki 重摄取漏网**: 如果在测试中发现任何 Fact 的来源指向了 Wiki 页面，必须立即停工重新审计路径过滤逻辑。

### 6. 一句话告诫
不要让“结构健康度”变成虚荣指标；如果底层事实本身是错误的，那么一个高度连接的网络也不过是设计精巧的系统性错误。
