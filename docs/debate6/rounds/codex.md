1. **Pick**: Option B.

2. **Rationale**: SQLite is safest when the durability boundary is one database file, one WAL, and one writer lock domain. Option A can be made to work, but only by turning the outbox drain into an attached-database protocol with hidden constraints the current spec does not model well enough: attachment lifecycle, alias management, `SQLITE_MAX_ATTACHED`, busy handling across multiple files, and recovery assumptions that now span several WAL files instead of one. In a 2-week Phase 0, that is the wrong place to spend complexity budget. Option B makes §1.6, §3/§3b, §8.4, §10.1, and the crash matrix materially simpler because the canonical drain becomes an ordinary single-DB transaction, and the reflect/drain race surface collapses to one lock domain instead of “ledger plus N outboxes.”

3. **Concrete failure mode your option survives that the other doesn't**: A user installs Compost in a host environment with 11 adapters over time: Claude Code, airi, openclaw, shell history, browser, calendar, email, git, notes, tasks, and one custom adapter. Under Option A, the daemon must either pre-`ATTACH` more databases than the default SQLite attachment limit allows, or dynamically attach/detach during drain. That is not just operationally annoying; it means the central durability path now depends on compile-time SQLite limits and on attach bookkeeping being correct under restart and hot-discovery conditions. Option B does not care how many adapters exist. Every adapter writes into the same `ledger.db`, and the transaction model remains unchanged.

4. **Concrete concession**: You give up the debate #1 per-adapter isolation invariant. A noisy or broken adapter can now create WAL contention for everyone else, and backup/export loses the nice property that adapter-local transient state lives in its own file. That is a real regression. I still take it, because Phase 0 is dominated by correctness of crash semantics, not by maximizing isolation between adapters that will usually write tiny append-only rows.

5. **Hook shim cold-start impact**: Option B probably still preserves the §3b.5 gate. Opening `ledger.db` does not read the whole database into memory; SQLite opens the file, reads schema pages, and appends to the WAL. On Apple silicon with `better-sqlite3`, a single-row append into a moderately sized WAL-mode database should still be within the p95 `< 30ms` budget if the shim stays otherwise minimal. The risk is real but measurable, and the spec already has the correct fallback: if `compost doctor --measure-hook` fails on reference hardware, ship the native Go hook binary in Phase 1. That is a cleaner fallback than baking attached-DB complexity into Phase 0 forever.

6. **Schema migration path**: Add a migration that creates `observe_outbox` inside `~/.compost/ledger.db` with an explicit `adapter` column and indexes on `(adapter, drained_at, seq)` plus the existing idempotency key. Update `compost hook` and the adapter SDK so all new appends target `ledger.db` directly. On daemon startup, run a one-time legacy import pass over `~/.compost/adapters/*/outbox.db`: for each undrained row, insert it into `ledger.observe_outbox` with the same `adapter`, `source_id`, `idempotency_key`, payload, timestamps, and a preserved legacy sequence field in metadata if needed for audit; drained rows can be ignored. After import, rename legacy `outbox.db` files to `outbox.db.migrated` so the system is single-path from then on. The important part is that after migration there is exactly one canonical transaction boundary left in the system.
