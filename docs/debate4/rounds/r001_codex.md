## Decision A
- Score 2/3
- One new failure mode NOT in the stress-test questions: cursor instability. If the SQL embeds `strftime('%s','now')` directly in the sort expression, page 1 and page 2 of the same query are ranked against different clocks, so rows drift across the cursor boundary and you get duplicates or misses. That also breaks auditability because the same query text no longer has a stable replay result (`<repo>/docs/reference-survey-airi.md:95-102`).
- One concrete schema/SQL change required: replace inline `now` in the ranking SQL with a bound `:as_of_unix_sec` parameter, and add `last_reinforced_at_unix_sec INTEGER NOT NULL DEFAULT (unixepoch())` so decay is computed from an explicit anchor column rather than overloading `created_at`.
- Rationale: stateless decay is correct as read-time math, but it has to be deterministic per request. With an explicit `as_of` anchor, A stays compatible with the locked `derivation_run` identity from debate #3 because ranking state remains outside rebuild identity (`<repo>/docs/debate3/synthesis.md:96-125`).

## Decision B
- Score 2/3
- One new failure mode NOT in the stress-test questions: NULL and scale contamination. On a heterogeneous corpus, some rows will have no emotional term, no access history, or no active context penalty. In SQL, one unguarded NULL can null out the whole expression, and `sqrt(access_count)` is numerically unbounded: at 10,000 recalls it contributes `100 * w3`, while cosine similarity and decay stay in `[0,1]`. AIRI only demonstrates a bounded two-factor formula, not the proposed seven-term mixture (`<repo>/docs/reference-survey-airi.md:81-102`).
- One concrete schema/SQL change required: define a bounded `score_v1` SQL expression where every term is `COALESCE(..., 0.0)` and the reinforcement term saturates, e.g. `MIN(1.0, LN(1 + access_count) / LN(1 + access_sat))` instead of raw `sqrt(access_count)`. Store the weights and transform version in a `ranking_profile` SQL table keyed by profile/context so bad rankings are replayable the same way policy rows are replayable in debate #3 (`<repo>/docs/debate3/synthesis.md:127-136`).
- Rationale: the direction is right. LycheeMem and the survey both point toward multi-factor retrieval rather than pure vector similarity (`<repo>/docs/reference-survey-memory-projects.md:93-108`, `<repo>/docs/reference-survey-memory-projects.md:257-264`). But “probabilistic” is not a license for unbounded arithmetic.

## Decision C
- Score 1/3
- One new failure mode NOT in the stress-test questions: shared page churn. Sensory rows with second-scale TTL and frequent deletes will fragment the same table pages that hold procedural rows with effectively infinite TTL. Even if `WHERE kind='semantic'` hits an index, the base table still pays the locality, VACUUM, and checkpoint cost of the high-churn tiers. That is a physical-storage failure, not just a logical-model concern.
- One concrete schema/SQL change required: if there is one logical `memories` base table, require kind-specific extension tables (`memory_episodic`, `memory_procedural`, etc.) keyed by `memory_id`, plus partial indexes on the base table for each hot query path such as `CREATE INDEX idx_mem_semantic_live ON memories(context_id, created_at DESC) WHERE kind='semantic' AND archived_at IS NULL;`.
- Rationale: AIRI does not actually collapse everything into one table; it splits fragments, episodic records, goals, and ideas into separate physical tables (`<repo>/docs/reference-survey-airi.md:60-65`). Debate #3 also just removed JSON-packed context metadata because `contexts TEXT` was already a scaling cliff (`<repo>/docs/debate3/synthesis.md:151-175`, `<repo>/docs/phase0-spec.md:174-191`). `WHERE kind='X'` can be an index hit, but only if you build the partial indexes explicitly; otherwise it is a scan.

## Decision D
- Score 1/3
- One new failure mode NOT in the stress-test questions: idempotency drift across hook retries. A short-lived `compost hook` subprocess has no durable per-adapter sequence state like the adapter SDK in `<repo>/docs/phase0-spec.md:280-321`. If Claude retries the same hook after a timeout and the event envelope does not deterministically reproduce the same `idempotency_key`, you create duplicate observations with no clean replay lineage.
- One concrete schema/SQL/code change required: do not replace the write protocol. Make `compost hook <event>` a Claude-specific adapter front door that writes into the same `observe_outbox` durability path from debate #1 and the current spec, then forwards via existing `compost.observe` (`<repo>/docs/debate/synthesis.md:27-36`, `<repo>/docs/phase0-spec.md:344-400`).
- Rationale: I do not see new evidence sufficient to reopen debate #1’s locked stdio MCP + outbox decision. Claude Code’s hook system still uses async task registration and later completion delivery via `queued_command` attachments (`<repo>/docs/reference-survey-claude-code-source.md:71-110`). That supports a Claude-specific ingress optimization, not a core protocol inversion.

## Composition check
- A/B/C/D do not compose cleanly as written. A says ranking is read-time derived state; B introduces reinforcement and repetition terms that require write-fed signals. AIRI already warns that per-access writes create lock contention on hot rows (`<repo>/docs/reference-survey-airi.md:432-462`).
- C makes B less stable because one cross-kind formula now operates over rows with different missing factors, TTL regimes, and metadata density. That creates NULL and calibration problems before retrieval quality problems.
- D leaks host-session semantics into the core model. Claude’s `async: true` contract is host-specific state (`<repo>/docs/reference-survey-claude-code-source.md:82-104`), while A is supposed to stay stateless and host-agnostic.
- The version that composes is: keep A as pure read-time math, feed B from an append-only retrieval-feedback log or batched rollup, reject C as a physical-table requirement, and treat D as an optional adapter shim over the debate #1 transport rather than a replacement.

## Final paragraph
If I had to ship Compost v2 tomorrow with only 2 of these 4 decisions adopted, I would take A and B, both with refinement. They improve retrieval behavior without reopening already-locked transport and schema boundaries. C fights debate #3’s normalization direction, and D fights debate #1’s durability decision without stronger evidence than one host’s hook implementation.
